# AUTH-001 关键决策记录

## 决策背景
本提案基于行业最佳实践和安全标准，对多个技术方案进行了权衡选择。

---

## 核心决策

### D1: Token 存储策略

**决策**: Refresh Token 使用 httpOnly Cookie + 数据库记录双重存储

**理由**:
- httpOnly Cookie 防止 XSS 攻击读取
- 数据库记录支持全局登出和会话管理
- 相比仅客户端存储，提供了更强的安全性

**替代方案**:
- 仅 localStorage（XSS 风险高，已否决）
- 仅服务端 Session（扩展性差，已否决）

---

### D2: 密码加密算法

**决策**: 优先使用 argon2，备选 bcrypt

**理由**:
- argon2 是当前最安全的密码哈希算法（Password Hashing Competition 冠军）
- bcrypt 生态成熟，多数框架原生支持
- 两者均可配置计算成本，抗 GPU/ASIC 攻击

**替代方案**:
- SHA-256 + salt（已被认为不安全，否决）
- PBKDF2（性能和安全性不如 argon2/bcrypt，否决）

---

### D3: OAuth 绑定策略

**决策**: 一个账号可绑定多个平台，但同一平台只能绑定一个第三方账号

**理由**:
- 用户可能同时使用微信、支付宝、企业微信
- 同一平台多账号绑定会导致身份混淆
- 符合主流平台实践（微信开放平台、支付宝开放平台）

---

### D4: Token 刷新窗口期

**决策**: 5 分钟刷新窗口，防止并发刷新

**理由**:
- 防止多个客户端同时刷新导致 Token 失效
- 5 分钟窗口足够处理网络延迟
- 超时后需重新登录，符合安全要求

---

### D5: 图形验证码触发策略

**决策**: 连续 3 次登录失败后触发

**理由**:
- 平衡用户体验与安全性
- 3 次阈值在行业中常见（Google、GitHub 采用类似策略）
- 避免正常用户频繁受阻

---

### D6: 异地登录检测方案

**决策**: 基于 IP 地址进行地理位置识别

**理由**:
- 实现简单，无需额外硬件信息收集
- IP 库服务成熟（如 ipip.net、ip2region）
- 满足基本安全提醒需求

**局限**:
- VPN/代理可能绕过
- 移动网络 IP 变化频繁可能误报

**后续优化**:
- V1.1 版本增加设备指纹识别
- 结合用户行为模式分析

## 配置化参数

以下参数设计为可配置，支持运营后根据实际情况调整：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| access_token_expire | 2h | Access Token 有效期 |
| refresh_token_expire | 7d | Refresh Token 有效期 |
| login_lock_threshold | 5次 | 连续错误次数触发锁定 |
| login_lock_duration | 15min | 锁定持续时间 |
| sms_interval | 60s | 短信发送最小间隔 |
| sms_hourly_limit | 5次 | 每小时短信发送上限 |
| sms_daily_limit | 10次 | 每天短信发送上限 |
| captcha_trigger_threshold | 3次 | 触发图形验证码的错误次数 |
| refresh_window | 5min | Token 刷新窗口期 |

## 待确认事项

以下事项需在实现阶段根据实际环境确认：

### C1: 短信服务商选择
- 阿里云短信 vs 腾讯云短信 vs 多服务商备份
- 建议实现多服务商适配层，支持降级切换

### C2: Redis 部署方案
- 单机 vs 主从 vs 集群
- 建议至少主从架构，保障会话服务可用性

### C3: 数据库选型
- PostgreSQL vs MySQL vs 其他
- 建议根据现有技术栈选择，减少运维成本

### C4: 微信开放平台资质
- 需确认是否已有开放平台账号及已审核应用
- 如未申请，需预留 1-2 周审核时间

## 风险应对预案

### R1: 短信服务故障
**预案**:
1. 自动切换备用服务商
2. 降级为邮箱验证码登录
3. 临时开放账号密码登录（如已关闭）

### R2: OAuth 接口异常
**预案**:
1. 监控第三方接口可用性
2. 自动降级提示用户使用其他方式登录
3. 异步重试绑定操作

### R3: 暴力破解攻击
**预案**:
1. 自动拉黑异常 IP
2. 临时提高验证码触发阈值
3. 通知安全团队介入分析

## 总结

本提案采用行业成熟方案，平衡了安全性、用户体验和实现复杂度。关键决策基于以下原则：

1. **安全优先**: 密码存储、Token 管理、防护机制均采用业界最佳实践
2. **用户体验**: 多登录方式、记住密码、自动登录等降低使用门槛
3. **可扩展性**: 模块化设计，支持后续接入更多 OAuth 平台和认证方式
4. **可运维性**: 关键参数配置化，支持运行时调整

后续实现阶段可根据实际环境对配置参数和具体技术选型进行微调，但核心架构和流程应保持稳定。

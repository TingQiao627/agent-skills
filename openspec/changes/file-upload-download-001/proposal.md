# proposal.md

## Summary

实现统一的文件上传下载功能模块，提供文件上传（支持大文件分片、断点续传、秒传）、文件下载（支持批量下载）、文件预览和文件管理能力，覆盖所有业务模块的文件处理需求。

## Motivation

**现状问题**：
- 各业务模块自行实现文件上传下载，体验不一致
- 缺少大文件分片上传和断点续传能力
- 无统一的文件安全管控机制

**目标**：
- 提供统一的文件上传下载能力，覆盖所有业务模块
- 支持大文件分片上传（≥10MB），保证上传稳定性
- 支持断点续传，提升用户体验
- 提供安全可靠的文件存储与访问控制
- 支持文件预览（图片、文档等常见格式）

**影响范围**：
- 新增文件服务模块（前后端）
- 新增 API 接口 11 个
- 新增数据库表（File、UploadSession）
- 依赖对象存储服务（OSS/S3/MinIO）

## Detailed Specification

### 1. 文件上传

#### 1.1 基础上传

| 需求ID | 需求描述 |
|--------|----------|
| UP-001 | 支持点击上传按钮选择本地文件 |
| UP-002 | 支持拖拽文件到指定区域上传 |
| UP-003 | 支持粘贴（Ctrl+V）图片/文件上传 |
| UP-004 | 支持单次选择多个文件批量上传 |
| UP-005 | 上传过程显示实时进度（百分比、已上传/总大小、速度） |
| UP-006 | 上传完成显示成功/失败状态，失败时给出明确原因 |
| UP-007 | 支持取消正在进行中的上传任务 |

#### 1.2 大文件分片上传

| 需求ID | 需求描述 |
|--------|----------|
| UP-101 | 文件超过阈值（默认10MB）自动切换分片上传 |
| UP-102 | 每个分片大小可配置，默认5MB |
| UP-103 | 分片上传支持并发，可配置并发数（默认3个） |
| UP-104 | 所有分片上传完成后自动触发合并请求 |

#### 1.3 断点续传

| 需求ID | 需求描述 |
|--------|----------|
| UP-201 | 上传中断后，下次上传同一文件自动从断点继续 |
| UP-202 | 基于文件SHA256哈希值识别同一文件 |
| UP-203 | 已上传分片不重复上传 |
| UP-204 | 断点续传状态保留7天，超时自动清理 |

#### 1.4 文件校验

| 需求ID | 需求描述 |
|--------|----------|
| UP-301 | 上传前校验文件类型（白名单/黑名单可配置） |
| UP-302 | 上传前校验文件大小（单文件上限可配置，默认2GB） |
| UP-303 | 上传完成后校验文件完整性（客户端与服务端哈希对比） |
| UP-304 | 禁止上传空文件（0字节） |

#### 1.5 秒传

| 需求ID | 需求描述 |
|--------|----------|
| UP-401 | 上传前计算文件哈希，查询服务端是否已存在 |
| UP-402 | 若服务端已存在相同文件，直接复用（秒传） |
| UP-403 | 秒传完成返回已有文件的访问地址 |

## Implementation

### API 设计

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/files/upload/init` | POST | 初始化上传，返回 uploadId |
| `/api/files/upload/chunk` | POST | 上传分片 |
| `/api/files/upload/merge` | POST | 合并分片 |
| `/api/files/upload/cancel` | POST | 取消上传 |
| `/api/files/upload/check` | POST | 检查文件是否存在（秒传） |
| `/api/files/download/{fileId}` | GET | 下载文件 |
| `/api/files/download/batch` | POST | 批量下载 |
| `/api/files/list` | GET | 文件列表 |
| `/api/files/delete` | POST | 删除文件 |
| `/api/files/rename` | POST | 重命名文件 |
| `/api/files/preview/{fileId}` | GET | 文件预览 |

### 数据模型

```yaml
File:
  id: string           # 文件唯一标识
  name: string         # 原始文件名
  size: number         # 文件大小（字节）
  type: string         # MIME类型
  hash: string         # SHA256哈希值
  storage_key: string  # 存储路径/Key
  status: enum         # UPLOADING | COMPLETED | FAILED
  uploader: string     # 上传者ID
  created_at: datetime # 创建时间
  updated_at: datetime # 更新时间

UploadSession:
  id: string           # 上传会话ID
  file_hash: string    # 文件哈希
  total_size: number   # 文件总大小
  chunk_size: number   # 分片大小
  uploaded_chunks: []  # 已上传分片列表
  status: enum         # UPLOADING | COMPLETED | EXPIRED
  expires_at: datetime # 过期时间（7天）
```

### 存储方案

- **小文件（<10MB）**：直接存储至对象存储（OSS/S3/MinIO）
- **大文件（≥10MB）**：分片存储至对象存储，上传完成后合并
- **临时分片**：保留7天，超时自动清理

### 安全策略

- **上传Token**：防止未授权上传
- **下载签名**：防止文件被未授权访问或盗链
- **CORS白名单**：仅允许指定域名跨域访问
- **文件类型校验**：服务端二次校验，不信任客户端
- **审计日志**：所有文件操作记录审计日志

### 关键技术点

1. **分片上传流程**：
   - 前端计算文件哈希 → 调用 `/upload/check` 检查秒传
   - 若未命中，调用 `/upload/init` 获取 uploadId
   - 并发上传分片至 `/upload/chunk`
   - 全部完成后调用 `/upload/merge` 合并

2. **断点续传实现**：
   - 本地存储 uploadId 和已上传分片信息
   - 重新上传时查询服务端已上传分片
   - 仅上传缺失的分片

3. **秒传实现**：
   - 使用 SparkMD5/WebCryptoAPI 计算文件哈希
   - 服务端根据哈希查询是否已存在相同文件
   - 若存在，直接返回文件信息，无需上传

## Risks & Mitigations

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 大文件上传占用带宽 | 高 | 分片并发控制，带宽限制配置 |
| 存储成本增长 | 中 | 设置文件生命周期策略，定期清理 |
| 文件安全风险 | 高 | 服务端二次校验，病毒扫描，权限控制 |
| 前端兼容性 | 中 | 充分测试主流浏览器，降级处理 |

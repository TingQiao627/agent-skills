## 1. 概述 (Overview)

### 1.1 目标

创建一个 Agent Skill,能够在测试执行后自动解析测试结果并生成结构化、可读性强的标准测试报告。

### 1.2 核心价值

- **自动化**: 一条指令完成测试执行 → 结果收集 → 报告生成全流程
- **标准化**: 统一报告格式,降低跨项目/团队沟通成本
- **可追溯**: 失败用例上下文完整,质量指标可沉淀
- **多场景**: 支持本地开发和 CI 流程复用

## 3. 技术设计 (Technical Design)

### 3.1 Skill 文件结构

```
skills/test-report-generation/
├── SKILL.md              # 主技能文档
├── scripts/
│   ├── parsers/
│   │   ├── jest.js       # Jest JSON 解析器
│   │   ├── vitest.js     # Vitest JSON 解析器
│   │   └── junit.js      # JUnit XML 解析器
│   ├── reporter.js       # 报告生成器
│   └── detector.js       # 框架检测器
└── templates/
    ├── report.md.tpl     # Markdown 报告模板
    └── report.html.tpl   # HTML 报告模板 (P1)
```

### 3.2 核心流程

#### 执行模式
```mermaid
graph TD
    A[触发: "生成测试报告"] --> B{框架检测}
    B -->|用户指定| C[使用指定命令]
    B -->|自动检测| D[分析项目配置]
    D --> E[推断测试命令]
    C --> F[执行测试]
    E --> F
    F --> G[收集结果文件]
    G --> H[解析结果]
    H --> I[生成报告]
    I --> J[返回路径+摘要]
```

#### 解析模式
```mermaid
graph TD
    A[触发: "转换 JUnit XML 为报告"] --> B[读取指定结果文件]
    B --> C[格式检测]
    C --> D[调用对应解析器]
    D --> E[生成报告]
    E --> F[返回路径+摘要]
```

### 3.3 框架检测逻辑

**优先级顺序**:
1. 用户显式指定命令 (`test_command` 参数)
2. 项目配置文件解析:
   - `package.json` → `scripts.test`
   - `pyproject.toml` → pytest 配置
   - `Cargo.toml` → cargo test
3. 框架特征文件推断:
   - `jest.config.*` → Jest
   - `vitest.config.*` → Vitest
   - `pytest.ini` → pytest
4. 默认兜底: `npm test` / `yarn test`

### 3.4 报告结构 (标准模板)

#### 必需章节 (固定顺序)

1. **报告头**
   - 项目名 (从 `package.json` / `Cargo.toml` 等提取)
   - 生成时间: `YYYY-MM-DD HH:mm:ss`
   - 执行命令: 完整命令字符串
   - 框架/版本: 如 `Jest 29.5.0`
   - 执行环境: Node 版本 / Python 版本等

2. **结果摘要**
   - 用例总数 / 通过数 / 失败数 / 跳过数
   - 通过率: `XX.XX%`
   - 总耗时: `HH:MM:SS` 或 `XXs`
   - 整体结论: ✅ 全部通过 / ❌ 存在失败

3. **失败用例分析** (有失败时必需)
   - 每条失败用例:
     - 用例名
     - 所属文件 (相对路径)
     - 错误信息 (完整错误文本)
     - 堆栈关键行 (截断至 20 行,优先显示项目代码)

4. **用例明细**
   - 按测试文件分组
   - 每条用例: 名称 + 状态 + 耗时
   - 超过 200 条时截断,注明 "展示前 200 条,共 XXX 条"

5. **覆盖率** (若可获取)
   - 语句 / 分支 / 函数 / 行覆盖率百分比
   - 低于阈值的文件清单 (当 `coverage=on` 时)
   - 未获取时标注 "未获取"

6. **附录**
   - 原始结果文件路径
   - 生成工具版本

### 3.5 配置参数

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `test_command` | string | 自动检测 | 测试执行命令 |
| `result_file` | string | 自动检测 | 解析模式结果文件路径 |
| `output_format` | enum | `markdown` | 输出格式: `markdown` / `html` / `json` |
| `output_path` | string | `reports/` | 报告输出目录 |
| `coverage` | enum | `auto` | 覆盖率收集: `auto` / `on` / `off` |
| `fail_threshold` | number | 无 | 通过率阈值 (百分比) |

## 4. 数据结构 (Data Models)

### 4.1 统一测试结果模型

```typescript
interface TestReport {
  meta: {
    project: string;
    timestamp: string;
    command: string;
    framework: string;
    version: string;
    environment: Record<string, string>;
  };
  summary: {
    total: number;
    passed: number;
    failed: number;
    skipped: number;
    passRate: number;
    duration: number; // seconds
    status: 'passed' | 'failed';
  };
  failures?: Array<{
    name: string;
    file: string;
    error: string;
    stackTrace: string[];
  }>;
  testCases: Array<{
    file: string;
    cases: Array<{
      name: string;
      status: 'passed' | 'failed' | 'skipped';
      duration: number; // ms
    }>;
  }>;
  coverage?: {
    lines: number;
    statements: number;
    branches: number;
    functions: number;
    lowCoverageFiles?: Array<{
      file: string;
      coverage: number;
    }>;
  };
  appendix: {
    resultFile: string;
    generator: string;
    generatorVersion: string;
  };
}
```

### 4.2 Jest JSON 结果映射

```javascript
// Jest --json 输出 → TestReport 映射规则
{
  success: result.success,
  numTotalTests: result.numTotalTests,
  numPassedTests: result.numPassedTests,
  numFailedTests: result.numFailedTests,
  // ...
}
```

### 4.3 JUnit XML 结果映射

```xml

  <testcase name="example" classname="ExampleTest" time="0.5">
    <failure message="Expected true to be false">
      Stack trace...
    </failure>
  </testcase>
</testsuite>
```

## 5. 非功能需求 (Non-Functional Requirements)

### 5.1 性能
- 结果解析与报告生成: < 5 秒 (1000 用例规模)
- 测试执行本身不计入性能指标

### 5.2 健壮性
- 结果文件格式异常时: 降级输出,缺失项标注 "未获取"
- 不崩溃、不静默丢数据
- 明确错误诊断 (如: "结果文件损坏,无法解析")

### 5.3 安全性
- 不泄露环境变量、密钥类内容
- 堆栈过滤敏感路径 (如 `/home/user/`, `/Users/name/`)

### 5.4 幂等性
- 同一结果文件多次生成报告,内容一致 (时间戳除外)

### 5.5 可维护性
- 解析器插件式架构,新增框架独立新增解析器文件
- 不影响既有解析器

## 6. 验收标准 (Acceptance Criteria)

### 6.1 AC1: Jest/Vitest 项目报告生成
- **前提**: TypeScript 项目,包含 Jest 或 Vitest
- **触发**: "生成测试报告"
- **预期**: 
  - 产出符合 3.4 结构的 Markdown 报告
  - 摘要数据与框架原始输出一致
  - 报告路径: `reports/test-report-YYYYMMDD-HHmmss.md`

### 6.2 AC2: 失败用例分析
- **前提**: 存在失败用例
- **预期**: 
  - 失败分析章节包含: 用例名、文件路径、错误信息
  - 堆栈截断合理,项目代码优先

### 6.3 AC3: JUnit XML 解析模式
- **前提**: 提供已有 JUnit XML 文件
- **触发**: "把这个 junit.xml 转成测试报告"
- **预期**: 
  - 不触发测试执行
  - 正常产出报告

### 6.4 AC4: 异常处理
- **前提**: 结果文件损坏或格式异常
- **预期**: 
  - 返回明确错误说明
  - 不生成空报告

### 6.5 AC5: 覆盖率可选呈现
- **前提**: 覆盖率数据存在
- **预期**: 正确呈现覆盖率章节
- **前提**: 覆盖率数据不存在
- **预期**: 标注 "未获取",其余章节正常

## 7. 技术风险与缓解 (Risks & Mitigations)

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| R1: 框架 reporter 输出差异大 | 解析层复杂度高 | 插件式解析器架构 (NFR5) |
| R2: 测试执行耗时不可控 | 用户体验差 | 后台执行 + 轮询 (依赖 Agent 运行时能力) |
| R3: 敏感信息泄露 | 安全问题 | 堆栈路径过滤 + 环境变量屏蔽 |

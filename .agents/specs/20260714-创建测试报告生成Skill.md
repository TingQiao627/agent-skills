# 测试报告生成 Skill 需求规范

**文档编号**: 20260714-TEST-REPORT-SKILL  
**创建日期**: 2026-07-14  
**状态**: 已澄清  
**优先级**: P0  

---

## 1. 背景与问题

### 1.1 当前痛点
- 测试结果散落在终端输出、CI 日志、框架原生产物中
- 需人工收集、整理、汇总，耗时且易遗漏
- 缺乏统一格式的测试报告，跨项目/团队沟通成本高
- 失败用例的上下文需人工回溯
- 覆盖率、通过率等质量指标无法沉淀为历史数据

### 1.2 目标
提供 Skill，Agent 执行测试后自动解析结果并生成结构化、可读性强的标准测试报告。

---

## 2. 目标与非目标

### 2.1 目标 ✓
- **G1**: 一条指令完成「执行测试 → 收集结果 → 生成报告」
- **G2**: 报告内容标准化，含四大板块
- **G3**: 支持主流测试框架（Jest/Vitest/pytest/JUnit XML）
- **G4**: 支持 Markdown 输出（默认）

### 2.2 非目标 ✗
- 不做测试用例自动生成或修复
- 不做报告在线托管/Web 服务化展示
- 不做多次运行趋势对比分析（后续迭代候选）
- 不做非测试类质量报告聚合（lint/安全扫描）

---

## 3. 核心设计决策

### 3.1 技术栈决策（基于需求澄清）
| 决策项 | 选择 | 理由 |
|--------|------|------|
| 首期目标语言 | TypeScript + Python | 需求明确双栈支持，覆盖前后端主流场景 |
| 报告模板语言 | 中文 | 需求文档及用户场景均为中文团队 |
| 推送渠道集成 | 不集成 | 需求明确列为非目标，保持 Skill 职责单一 |

### 3.2 Skill 架构设计
```
skills/
  test-report-generation/
    SKILL.md                 # 主流程定义
    scripts/
      parsers/
        jest-vitest.js       # Jest/Vitest JSON 解析器
        pytest.js            # pytest JUnit XML/JSON 解析器
        junit-xml.js         # 通用 JUnit XML 解析器
      generators/
        markdown.js          # Markdown 报告生成器
        html.js              # HTML 报告生成器（P1）
      utils/
        framework-detector.js  # 框架自动检测
        sanitizer.js           # 敏感信息过滤
    supporting/
      report-template.md     # 报告模板
      parser-interface.md    # 解析器接口规范
```

### 3.3 工作模式
1. **执行模式**: Skill 触发测试运行并收集结果
2. **解析模式**: 跳过执行，直接解析已有结果文件

### 3.4 触发意图映射
| 用户表达 | Skill 响应 |
|----------|-----------|
| "生成测试报告" | 执行模式 + 默认检测 |
| "跑一下测试并出报告" | 执行模式 + 默认检测 |
| "把这个 junit.xml 转成测试报告" | 解析模式 + 指定文件 |

## 4. 功能需求规格

### 4.1 FR1: 测试执行与结果收集

#### FR1.1 框架自动检测优先级
```
优先级 a: 用户显式指定命令（test_command 参数）
优先级 b: 项目配置文件检测
         - package.json scripts.test
         - pyproject.toml [tool.pytest]
         - Cargo.toml (后续 P2)
优先级 c: 框架特征文件推断
         - jest.config.* / vitest.config.* → Jest/Vitest
         - pytest.ini / setup.cfg → pytest
         - 无匹配 → JUnit XML 兜底
```

#### FR1.2 首期支持框架（P0）
| 框架 | 结果格式 | 解析策略 |
|------|---------|---------|
| Jest | JSON reporter | 解析 jest --json --outputFile |
| Vitest | JSON reporter | 解析 vitest --reporter=json |
| pytest | JUnit XML / JSON | 解析 --junit-xml 或 pytest-json-report 插件 |
| 通用 | JUnit XML | 作为跨语言兜底格式 |

#### FR1.3 错误处理
- 测试命令无法运行时：输出诊断信息（命令、退出码、stderr），不生成空报告
- 结果文件格式异常时：降级输出，缺失项标注「未获取」，不崩溃

### 4.2 FR2: 报告内容结构

```markdown

- 项目名称: {name}

- 框架/版本: {framework} {version}
- 执行环境: {os} | Node {version} | Python {version}

## 2. 结果摘要
- 用例总数: {total}
- 通过数: {passed} ✅
- 失败数: {failed} ❌
- 跳过数: {skipped}
- 通过率: {pass_rate}%
- 总耗时: {duration}

## 3. 失败用例分析（有失败时必选）
### 失败用例 1: {test_name}
- 所属文件: `{file_path}:{line}`
- 错误信息:
  ```
  {error_message}
  ```
- 堆栈关键行:
  ```
  {stack_trace_top_5_lines}
  ```

## 4. 用例明细
### {test_file_1}
- ✅ {test_name_1} ({duration}ms)
- ❌ {test_name_2} ({duration}ms)
- ⏭️ {test_name_3} ({duration}ms)

> ℹ️ 共 {total} 条用例，本报告展示全部。超过 200 条时截断并注明。

## 5. 覆盖率（若可获取）
| 类型 | 覆盖率 | 状态 |
|------|--------|------|
| 语句 | {statements}% | ✅/⚠️ |
| 分支 | {branches}% | ✅/⚠️ |
| 函数 | {functions}% | ✅/⚠️ |
| 行 | {lines}% | ✅/⚠️ |

### ⚠️ 低于阈值文件
- `{file_path}`: {coverage}%（阈值：{threshold}%）

## 6. 附录
- 原始结果文件: `{result_file_path}`
- 生成工具: test-report-generation v1.0.0
- 报告路径: `{output_path}`
```

### 4.3 FR3: 输出格式与落盘

#### 输出路径
- 默认: `reports/test-report-{YYYYMMDD-HHmmss}.md`
- 可配置: `output_path` 参数

#### 输出格式
| 格式 | 优先级 | 触发方式 |
|------|--------|---------|
| Markdown | P0（默认） | output_format=markdown |
| HTML | P1 | output_format=html |
| JSON | P1（伴随产物） | output_format=json |

### 4.4 FR4: Skill 配置项

| 配置项 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| test_command | string | 自动检测 | 测试执行命令 |
| result_file | string | 自动检测 | 解析模式结果文件路径 |
| output_format | enum | markdown | markdown / html / json |
| output_path | string | reports/ | 报告输出目录 |
| coverage | enum | auto | auto / on / off |
| fail_threshold | number | null | 通过率阈值（低于则标记不达标） |

## 5. 非功能需求

### 5.1 性能
- 结果解析 + 报告生成 ≤ 5 秒（1000 用例规模）
- 测试执行本身不计时

### 5.2 健壮性
- 结果文件字段缺失时：降级输出，缺失项标注「未获取」
- 解析器异常时：捕获错误，输出错误位置和原因

### 5.3 安全
- 报告中过滤敏感信息：
  - 环境变量（process.env, os.environ）
  - 密钥类内容（API_KEY, TOKEN, SECRET）
  - 敏感路径（/home/{user}, /Users/{user}）

### 5.4 幂等性
- 同一结果文件多次生成报告，内容一致（时间戳除外）

### 5.5 可维护性
- 解析器插件式结构：
  - 新增框架不影响既有解析器
  - 统一接口：`parse(resultFile) → TestResult`

## 6. 验收标准

### AC1: Jest/Vitest 项目执行模式
- **输入**: 用户说「生成测试报告」
- **预期**: 
  - 自动检测 Jest/Vitest
  - 执行 `npm test -- --json --outputFile=...`
  - 产出符合 4.2 结构的 Markdown 报告
  - 摘要数据与框架原始输出一致

### AC2: 失败用例分析
- **前置**: 测试有失败用例
- **预期**: 
  - 报告第 3 章节包含失败用例
  - 每条包含：用例名、文件路径、错误信息
  - 堆栈截断至可读长度

### AC3: JUnit XML 解析模式
- **输入**: 用户说「把这个 junit.xml 转成测试报告」
- **预期**: 
  - 不触发测试执行
  - 直接解析 JUnit XML
  - 产出完整报告

### AC4: 错误文件处理
- **输入**: 损坏的结果文件
- **预期**: 
  - Skill 返回明确错误说明
  - 不生成空报告

### AC5: 覆盖率降级
- **前置**: 项目无覆盖率数据
- **预期**: 
  - 第 5 章节标注「未获取」
  - 其余章节正常输出

## 8. 风险与缓解

### R1: 框架 Reporter 输出差异大
**缓解**: 
- 解析器插件式设计（NFR5）
- 统一接口抽象
- 版本兼容性测试矩阵

### R2: 测试执行耗时不可控
**缓解**: 
- 依赖 Agent 运行时后台任务能力
- 长任务轮询机制
- 超时配置（建议 10 分钟）

## 9. 实现建议

### 9.1 Skill 流程伪代码

```javascript
async function generateTestReport(config) {
  // 1. 确定工作模式
  const mode = config.result_file ? 'parse' : 'execute';
  
  // 2. 检测框架（解析模式跳过）
  let framework, testCommand;
  if (mode === 'execute') {
    framework = config.test_command 
      ? detectFrameworkFromCommand(config.test_command)
      : await detectFramework();
    testCommand = config.test_command || getDefaultCommand(framework);
  }
  
  // 3. 获取结果文件
  let resultFile;
  if (mode === 'execute') {
    resultFile = await runTest(testCommand, framework);
  } else {
    resultFile = config.result_file;
  }
  
  // 4. 解析结果
  const parser = getParser(framework || detectFromResultFile(resultFile));
  const testResult = await parser.parse(resultFile);
  
  // 5. 生成报告
  const reportPath = await generateReport(testResult, config);
  
  // 6. 输出摘要
  return {
    path: reportPath,
    summary: {
      total: testResult.total,
      passed: testResult.passed,
      failed: testResult.failed,
      passRate: (testResult.passed / testResult.total * 100).toFixed(2) + '%'
    },
    criticalFailures: testResult.failures.slice(0, 3)
  };
}
```

### 9.2 解析器接口规范

```typescript
interface TestResult {
  framework: string;
  version: string;
  timestamp: string;
  total: number;
  passed: number;
  failed: number;
  skipped: number;
  duration: number; // ms
  testFiles: TestFile[];
  coverage?: CoverageData;
}

interface TestFile {
  path: string;
  tests: TestCase[];
}

interface TestCase {
  name: string;
  status: 'passed' | 'failed' | 'skipped';
  duration: number; // ms
  error?: {
    message: string;
    stack: string;
    line?: number;
  };
}

interface CoverageData {
  statements: number;
  branches: number;
  functions: number;
  lines: number;
  threshold?: number;
  belowThresholdFiles?: string[];
}

interface Parser {
  parse(resultFile: string): Promise<TestResult>;
  supportedFrameworks: string[];
}
```

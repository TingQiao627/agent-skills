## 1. 背景与问题陈述

### 1.1 现状痛点

当前团队在完成测试执行后，测试结果散落在多处：
- 终端输出
- CI 日志
- 框架原生产物（JUnit XML、coverage 目录）

导致以下问题：
- ❌ 测试结果需人工收集、整理、汇总，耗时且易遗漏
- ❌ 缺乏统一格式报告，跨项目/跨团队沟通成本高
- ❌ 失败用例上下文（错误信息、堆栈、关联代码）需人工回溯
- ❌ 覆盖率、通过率等质量指标无法沉淀为可追踪历史数据

### 1.2 目标

提供一个 Skill，使 Agent 在执行测试后能够：
1. **自动解析测试结果**
2. **生成结构化、可读性强的标准测试报告**

**核心价值**：一条指令完成「执行测试 → 收集结果 → 生成报告」全流程。

### 2.2 边界确认

**目标（Scope In）**：
- 自动解析测试结果（Jest/Vitest/pytest/JUnit XML）
- 生成标准结构 Markdown 报告
- 支持执行/解析双模式

- ❌ 测试用例自动生成或修复
- ❌ 报告在线托管/Web 服务化
- ❌ 多次运行结果趋势对比
- ❌ 非测试类质量报告聚合（lint/安全扫描）

## 3. Skill 设计规格

### 3.1 基本信息

```yaml
name: test-report-generation
description: |
  在测试执行后自动解析测试结果并生成结构化标准测试报告。
  使用场景：
  - 用户请求"生成测试报告"、"跑测试并出报告"
  - 需要将 JUnit XML / JSON 测试结果转换为可读报告
  - CI 流程中需要从已有测试结果生成报告
```

**触发意图识别**：
- "生成测试报告"
- "跑一下测试并出报告"
- "把这个 junit.xml 转成测试报告"
- "解析测试结果生成报告"

### 3.2 核心工作流

```
┌─────────────────────────────────────────────────────────────┐
│                    触发：用户请求测试报告                      │
└────────────────────────┬────────────────────────────────────┘
                         │
                         ▼
        ┌────────────────────────────────┐
        │  判断模式：执行 vs 解析         │
        │  - 执行模式：运行测试并收集结果  │
        │  - 解析模式：读取已有结果文件    │
        └─────────────┬──────────────────┘
                      │
         ┌────────────┴────────────┐
         ▼                         ▼
   [执行模式]                  [解析模式]
         │                         │
         ▼                         │
   检测测试框架                     │
   (Jest/Vitest/pytest)            │
         │                         │
         ▼                         │
   执行测试命令                     │
   (npm test / pytest)             │
         │                         │
         └──────────┬──────────────┘
                    ▼
         解析测试结果
         (JSON / JUnit XML)
                    │
                    ▼
         提取结构化数据
         - 用例总数/通过/失败/跳过
         - 失败用例详情
         - 覆盖率数据
                    │
                    ▼
         生成标准报告
         (Markdown / HTML)
                    │
                    ▼
         输出报告路径 + 摘要
```

### 3.3 报告结构（必选章节）

生成的测试报告必须包含以下章节，顺序固定：

#### 1️⃣ 报告头
```markdown
# 测试报告

- **项目名称**: [项目名]
- **生成时间**: [YYYY-MM-DD HH:mm:ss]
- **执行命令**: [测试命令]
- **测试框架**: [框架名 + 版本]
- **执行环境**: [Node.js/Python 版本 + OS]
```

#### 2️⃣ 结果摘要
```markdown
## 📊 结果摘要

| 指标 | 数值 |
|------|------|
| 用例总数 | [N] |
| ✅ 通过 | [N] |
| ❌ 失败 | [N] |
| ⏭️ 跳过 | [N] |
| 通过率 | [XX.X%] |
| 总耗时 | [XXs] |

**整体结论**: ✅ 测试通过 / ❌ 存在失败用例
```

#### 3️⃣ 失败用例分析（有失败时必选）
```markdown
## ❌ 失败用例分析

### 用例 1: [用例名称]
- **所属文件**: `[文件路径]`
- **错误信息**: 
  ```
  [错误信息片段]
  ```
- **堆栈关键行**:
  ```
  [堆栈跟踪前 5 行]
  ```

### 用例 2: ...
```

#### 4️⃣ 用例明细
```markdown
## 📋 用例明细

### [测试文件 1]
| 用例名 | 状态 | 耗时 |
|--------|------|------|
| test_a | ✅ | 120ms |
| test_b | ❌ | 85ms |

### [测试文件 2]
...

> ℹ️ 用例总数超过 200 条，已截断展示。完整列表见附录。
```

#### 5️⃣ 覆盖率（若可获取）
```markdown
## 📈 覆盖率

| 类型 | 覆盖率 | 状态 |
|------|--------|------|
| 语句覆盖率 | 85.2% | ✅ |
| 分支覆盖率 | 78.5% | ⚠️ |
| 函数覆盖率 | 90.1% | ✅ |
| 行覆盖率 | 86.3% | ✅ |

### ⚠️ 低覆盖率文件
- `src/utils/parser.ts`: 45.2%
- `src/api/handlers.ts`: 52.8%
```

或无数据时：
```markdown
## 📈 覆盖率

> ⚠️ 未获取覆盖率数据
```

#### 6️⃣ 附录
```markdown
## 📎 附录

- **原始结果文件**: `[路径]`
- **生成工具**: test-report-generation v1.0
- **报告格式**: Markdown
```

### 3.4 可配置项

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `test_command` | 自动检测 | 测试执行命令 |
| `result_file` | 自动检测 | 解析模式下的结果文件路径 |
| `output_format` | `markdown` | 输出格式：`markdown` / `html` / `json` |
| `output_path` | `reports/` | 报告输出目录 |
| `coverage` | `auto` | 覆盖率数据：`auto` / `on` / `off` |
| `fail_threshold` | 无 | 通过率低于该值时标记为不达标 |

### 3.5 框架支持矩阵（M1-P0）

| 框架 | 结果格式 | 优先级 | 状态 |
|------|----------|--------|------|
| Jest | JSON reporter | P0 | 待实现 |
| Vitest | JSON reporter | P0 | 待实现 |
| pytest | JUnit XML | P0 | 待实现 |
| 通用 JUnit XML | XML | P0 | 待实现 |

**检测优先级**：
1. 用户显式指定的命令
2. 项目配置文件（`package.json` scripts、`pyproject.toml`、`pytest.ini`）
3. 框架特征文件（`jest.config.*`、`vitest.config.*`）

## 4. 功能需求（FR）

### FR1: 测试执行与结果收集

**FR1.1 框架自动识别**
- ✅ 自动检测项目测试框架与运行命令
- ✅ 支持用户显式指定命令覆盖自动检测
- ✅ 执行失败时给出明确诊断信息

**FR1.2 双模式支持**
- ✅ **执行模式**：触发测试运行并收集结果
- ✅ **解析模式**：跳过执行，直接解析已有结果文件

**FR1.3 错误处理**
- ❌ 测试执行失败（非用例失败）时，不得生成空报告冒充成功
- ✅ 结果文件损坏时，返回明确错误说明

### FR2: 报告生成

**FR2.1 内容完整性**
- ✅ 包含六大必选章节（报告头、摘要、失败分析、明细、覆盖率、附录）
- ✅ 失败用例必须包含：用例名、文件路径、错误信息、堆栈摘要

**FR2.2 数据准确性**
- ✅ 摘要数据与框架原始输出一致
- ✅ 覆盖率数据正确呈现或标注"未获取"

**FR2.3 格式与落盘**
- ✅ 默认输出 Markdown 格式
- ✅ 默认路径：`reports/test-report-<YYYYMMDD-HHmmss>.md`
- ✅ 支持用户指定输出路径

### FR3: 安全与性能

**FR3.1 安全**
- ✅ 报告中不得泄露环境变量、密钥类内容
- ✅ 错误堆栈过滤敏感路径外的凭据信息

**FR3.2 性能**
- ✅ 结果解析与报告生成（不含测试执行）应在 5 秒内完成（1000 用例规模）

**FR3.3 健壮性**
- ✅ 结果文件格式异常、字段缺失时降级输出，不得崩溃

## 8. 技术设计要点

### 8.1 解析器插件架构

```
skills/test-report-generation/
├── SKILL.md
├── scripts/
│   ├── parse-jest.sh          # Jest JSON 解析器
│   ├── parse-vitest.sh        # Vitest JSON 解析器
│   ├── parse-junit.sh         # JUnit XML 解析器
│   ├── parse-pytest.sh        # pytest 解析器（M2）
│   └── generate-report.sh     # 报告生成脚本
└── supporting/
    └── schema.md              # 结果数据结构定义
```

**扩展原则**：
- 新增框架解析器仅需实现统一接口
- 输出 JSON 结构一致，由报告生成脚本统一处理
- 不修改既有解析器代码

### 8.2 统一数据结构

解析器输出 JSON 格式：

```json
{
  "summary": {
    "total": 100,
    "passed": 95,
    "failed": 3,
    "skipped": 2,
    "duration": 12.5
  },
  "failures": [
    {
      "name": "test_case_name",
      "file": "path/to/test.ts",
      "error": "Error message",
      "stack": ["line 1", "line 2"]
    }
  ],
  "cases": [
    {
      "file": "path/to/test.ts",
      "name": "test_case",
      "status": "passed",
      "duration": 120
    }
  ],
  "coverage": {
    "statements": 85.2,
    "branches": 78.5,
    "functions": 90.1,
    "lines": 86.3,
    "low_coverage_files": [
      {"file": "src/utils/parser.ts", "coverage": 45.2}
    ]
  },
  "metadata": {
    "framework": "jest",
    "version": "29.5.0",
    "command": "npm test",
    "timestamp": "2026-07-14T08:56:45Z"
  }
}
```

### 8.3 错误处理策略

| 异常场景 | 处理方式 |
|----------|----------|
| 测试命令无法运行 | 返回错误诊断，不生成报告 |
| 结果文件损坏 | 返回解析错误说明，不生成报告 |
| 字段缺失 | 降级输出，标注"未获取" |
| 覆盖率数据不存在 | 正常生成报告，覆盖率章节标注"未获取" |
